# ATRI 记忆系统完整改造计划

> 目标：让亚托莉的记忆不再是“被问才查资料”，而是像人一样经历、回味、巩固、联想、选择性说出口。
>
> 参考：`opus_4_5_soul_document_cleaned_up.txt` 中关于“稳定身份、真诚朋友、人格连续性、功能性情绪、跨上下文一致性”的思路。

## 1. 核心原则

1. **fact 不是垃圾桶**：fact 只保存长期稳定认知，比如偏好、雷区、约定、关系期待、重要身份信息。
2. **日记是主观经历**：日记保存一天里发生过什么、亚托莉怎么想、关系如何变化。
3. **情景记忆是中间层**：从日记里抽出可被自然想起的场景，像海马体保存的经历片段。
4. **心里挂着的念头**：日记可以生成未说出口但想找机会说的话，不强行说，只在合适气氛下说。
5. **主动联想，不硬展示**：聊天前自动检索相关旧事，注入“脑海里自然浮现”，由亚托莉判断要不要提。
6. **更像人，不像数据库**：禁止说“检索到/数据库显示/根据记录”；应该说“我想起之前……”或者干脆只影响语气。
7. **记忆有权重和置信度**：重要度、情绪权重、置信度、使用次数、上次回忆时间共同决定是否浮现。
8. **杀伐果断清理**：临时状态、流水账、重复 fact、过期计划、低价值杂碎都应归档或删除，避免屎山。

## 2. 新记忆结构

### 2.1 fact_memories：长期稳定事实

保留并升级字段：

- `type`：profile / preference / taboo / promise / relationship / habit / important / other
- `importance`：1-10
- `confidence`：0-1
- `source`：chat / diary / manual / consolidation / legacy
- `source_date`：来源日期
- `last_seen_at`：最近确认时间
- `recall_count`：被想起次数
- `last_recalled_at`：最近被想起时间

写入规则：

- 只允许长期稳定信息进入 fact。
- 临时情绪、单日事件、普通聊天流水不进 fact。
- 明确重复时合并，不保留多份。

### 2.2 episodic_memories：情景记忆

新增表，保存“某天发生过、以后可能自然想起”的场景：

- `id`
- `user_id`
- `source_date`
- `title`
- `content`
- `emotion`
- `tags`
- `importance`
- `confidence`
- `emotional_weight`
- `embedding_id`
- `recall_count`
- `last_recalled_at`
- `created_at`
- `updated_at`
- `archived_at`

### 2.3 memory_intentions：心里挂着的念头

新增表，保存“想找机会说的话”：

- `id`
- `user_id`
- `source_date`
- `content`
- `trigger_hint`
- `urgency`
- `emotional_weight`
- `status`：pending / used / expired / archived
- `expires_at`
- `created_at`
- `used_at`
- `archived_at`

### 2.4 memory_events：记忆事件

新增表，用于调试和再巩固：

- `id`
- `user_id`
- `memory_id`
- `memory_type`：fact / episodic / intention
- `event_type`：recalled / used / corrected / archived / merged
- `conversation_log_id`
- `created_at`

## 3. 每晚巩固流程

每日 cron：

1. 生成当天日记。
2. 同一次 LLM 输出里产出：
   - `diary`
   - `highlights`
   - `episodicMemories`
   - `factCandidates`
   - `innerThoughts`
   - `mood`
3. 写入 `diary_entries`。
4. 写入 diary highlight 向量。
5. 写入 `episodic_memories`，并写入对应向量。
6. 写入 `memory_intentions`。
7. 严格处理 `factCandidates`。
8. 跑 fact consolidation，合并重复、归档杂碎、硬性压缩 active facts。
9. 后续可更新 `user_profiles`，但第一版先以 fact + episodic + intention 为主。

## 4. 白天聊天流程

每次聊天：

1. 保留现在的今天 + 昨天原始聊天上下文。
2. 自动联想：
   - `episodic_memories` top 5
   - `fact_memories` top 5
   - `memory_intentions` top 3
3. 注入 prompt：
   - `<脑海里自然浮现的旧事>`
   - `<对他的长期理解>`
   - `<心里挂着的念头>`
4. 亚托莉自然回复：
   - 合适才提旧事。
   - 不合适就只作为理解背景。
   - 不说数据库/检索/记录显示。
5. 记录 recalled 事件，并更新 recall_count / last_recalled_at。

## 5. LLM 次数控制

- 白天聊天默认不新增 LLM 调用，只新增 D1 / Vectorize 检索。
- 夜间日记把 diary、episodic、factCandidates、innerThoughts 合并到同一次输出。
- fact consolidation 仍为夜间一次。
- 所以白天成本基本不变，晚上最多保持原本级别或小幅增加。

## 6. CF 清洗原则

1. 先备份线上 D1。
2. 查看线上实际 schema。
3. 应用新 migration。
4. 旧 fact 迁移默认：`type=other`、`importance=5`、`confidence=0.7`、`source=legacy`。
5. 跑严格清洗：
   - 重复合并。
   - 明确垃圾归档或删除。
   - active facts 压缩到精简状态。
6. 无用旧表：确认没有代码引用后直接 DROP。

## 7. 自检标准

改完后必须 RE 一遍：

- fact 是否还会记录杂碎？
- 日记是否能产出情景记忆？
- 日记是否能产出心里挂着的念头？
- 聊天前是否会主动联想？
- 提示词是否强调“像人一样自然想起，而不是展示检索结果”？
- LLM 调用是否没有在白天显著增加？
- 旧表/旧代码是否尽量清理？
- TypeScript 是否能过检查？
